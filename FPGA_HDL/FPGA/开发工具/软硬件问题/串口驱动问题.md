# CH340串口驱动在Linux上的问题解决笔记

## 问题现象
1. `lsusb`能看到CH340设备：`Bus 001 Device 009: ID 1a86:7523 QinHeng Electronics CH340 serial converter`
2. 但找不到设备文件：`ls: cannot access '/dev/ttyUSB*': No such file or directory`
3. 实际设备被`brltty`（盲文显示服务）占用

## 根本原因
CH340驱动已正确加载，但`brltty`服务抢先占用了串口设备，导致`ch341`驱动无法创建设备文件。

## 诊断路径

### 1. 检查驱动状态
```bash
# 检查CH340驱动是否可用
modinfo ch341

# 查看内核消息（插入设备时）
sudo dmesg -w
# 关键错误信息：
# usb 1-2: usbfs: interface 0 claimed by ch341 while 'brltty' sets config #1
```

### 2. 检查设备识别
```bash
# 查看USB设备
lsusb
# 查看所有可能的串口设备
ls /dev/tty* | grep -E "(USB|ACM|S[0-9])"
# 查看内核驱动加载情况
lsmod | grep ch341
```

## 解决方案

### 方案1：禁用brltty（推荐）
```bash
# 停止brltty服务
sudo systemctl stop brltty

# 禁用开机自启
sudo systemctl disable brltty

# 阻止自动加载
echo 'blacklist brltty' | sudo tee /etc/modprobe.d/blacklist-brltty.conf

# 重新插拔CH340设备
```

### 方案2：卸载brltty
```bash
# 如果不需要盲文支持
sudo apt remove brltty
# 或完全卸载
sudo apt purge brltty
```

### 方案3：配置brltty忽略CH340
```bash
# 编辑配置文件
sudo nano /etc/brltty.conf
# 添加：
# usb-ignore 1a86:7523
# 保存后重启服务
sudo systemctl restart brltty
```

## 验证和测试

### 1. 验证设备创建
```bash
# 停止brltty后重新插入设备
sudo dmesg | tail -20
# 应该看到正常信息：
# ch341 1-2:1.0: ch341-uart converter detected
# usb 1-2: ch341-uart converter now attached to ttyUSB0

# 检查设备文件
ls -la /dev/ttyUSB0
# 正常输出：crw-rw---- 1 root dialout 188, 0 Apr  7 10:30 /dev/ttyUSB0
```

### 2. 解决权限问题
```bash
# 临时解决
sudo chmod 666 /dev/ttyUSB0

# 永久解决：将用户加入dialout组
sudo usermod -a -G dialout $USER
sudo usermod -a -G tty $USER
# 需要重新登录生效

# 或创建udev规则
sudo nano /etc/udev/rules.d/99-ch340.rules
```
添加内容：
```bash
# CH340/CH341 串口设备
SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", MODE="0666", GROUP="dialout", SYMLINK+="ttyCH340"
SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7522", MODE="0666", GROUP="dialout", SYMLINK+="ttyCH340"
```
生效：
```bash
sudo udevadm control --reload-rules
sudo udevadm trigger
```

## 快速诊断脚本
```bash
#!/bin/bash
echo "=== CH340串口诊断脚本 ==="
echo ""
echo "1. USB设备:"
lsusb | grep -i "1a86\|ch340" || echo "未找到CH340设备"
echo ""
echo "2. 内核消息:"
dmesg | grep -i "ch34\|tty\|usb.*serial" | tail -10
echo ""
echo "3. 设备文件:"
for dev in /dev/ttyUSB* /dev/ttyACM*; do
    [ -e "$dev" ] && echo "$dev: $(ls -la $dev)"
done
echo ""
echo "4. 服务状态:"
systemctl status brltty --no-pager | grep -A3 "Active:"
echo ""
echo "5. 用户组:"
groups | grep -E "dialout|tty" || echo "用户不在dialout/tty组"
```

## 注意事项

1. **驱动位置**：CH340驱动在Linux内核中，不需要Windows驱动
2. **设备名**：可能是`/dev/ttyUSB0`或`/dev/ttyUSB1`等
3. **波特率**：必须与设备设置匹配（常见115200）
4. **权限**：用户必须在`dialout`组或有读写权限
5. **brltty冲突**：这是Ubuntu/Debian常见问题

## 扩展知识

### 手动加载驱动
```bash
# 如果驱动未自动加载
sudo modprobe ch341
sudo modprobe usbserial

# 手动绑定设备
echo "1a86 7523" | sudo tee /sys/bus/usb/drivers/ch341/new_id
```

### 查看详细USB信息
```bash
# 查看USB拓扑
lsusb -t
# 查看详细设备信息
lsusb -v -d 1a86:7523
# 查看内核USB设备
sudo cat /sys/kernel/debug/usb/devices | grep -A20 "1a86"
```

### Python测试脚本
```python
import serial
import time

# 测试串口通信
ser = serial.Serial(
    port='/dev/ttyUSB0',
    baudrate=115200,
    bytesize=8,
    parity='N',
    stopbits=1,
    timeout=1
)

# 发送测试数据
test_data = [0x00, 0xFF, 0x55, 0xAA]
for data in test_data:
    ser.write(bytes([data]))
    time.sleep(0.1)
    print(f"发送: 0x{data:02X}")

ser.close()
```

## 故障排除流程
```
1. 插入设备 → 查看dmesg → 有brltty错误？ → 停止brltty
2. 重新插入 → 查看/dev/ttyUSB* → 存在？ → 检查权限
3. 权限正确？ → 测试minicom → 成功？
4. 失败 → 检查波特率 → 检查接线 → 检查FPGA程序
```

记住这个流程，下次遇到类似问题可以快速定位解决。
