# Vivado 在 Ubuntu 下识别 FPGA 开发板问题排查与解决笔记

## 一、问题核心：权限不足导致通信阻断
**问题现象**：Ubuntu 22.04 系统中，`lsusb` 可识别 JTAG 下载器硬件，但 Vivado 的 Hardware Manager 提示“找不到 FPGA”或“No device found”。

**根本原因**：Linux 系统默认将 USB 编程设备（如 Digilent/JTAG 下载器）的访问权限仅分配给 root 用户。普通用户运行的 Vivado 因**缺乏对 USB 设备文件的写入权限**，无法向下载器发送命令，导致 JTAG 通信链无法建立，进而误判为 FPGA 不存在。

---

## 二、核心概念：Linux 文件权限与用户/组

### 1. 权限三元组
```
crw-rw-r-- 1 root root 189, 4 Mar 31 10:30 /dev/bus/usb/001/005
```
- **所有者权限** (`rw-`)：文件所有者（root）可读写
- **组权限** (`rw-`)：文件所属组成员（root 组）可读写
- **其他用户权限** (`r--`)：其他所有用户（包括你的普通账户）**仅可读**

### 2. 用户与组的关系
- 你的用户（如 `mountaincat`）通常不在 `root` 组中
- 对属于 `root:root` 的设备文件，你属于“其他用户”，只有读(`r`)权限
- **JTAG 通信需要双向读写**：Vivado 必须能向设备**写入命令**并读取响应

---

## 三、不同连接方式的差异对比

| 连接类型 | 识别级别 | 权限要求 | 工作原理 | 常见表现 |
|---------|---------|---------|---------|---------|
| **Type-C 转 USB 线** | 通常识别为串口设备 (`/dev/ttyUSB*`) | 用户只需在 `dialout` 组 | 开发板内置 USB 转 JTAG/UART 芯片，主机通过标准串口驱动通信 | 较易识别，权限门槛低 |
| **独立 JTAG 下载器** (如 FT232H) | 识别为 USB 编程设备 (`/dev/bus/usb/...`) | 需要直接 USB 端点访问权限 | Vivado 通过 libusb 直接与下载器芯片通信，进行底层 JTAG 协议转换 | 权限要求严格，需特殊配置 |

**关键区别**：Type-C 线可能仅用于数据传输（如 UART），而独立 JTAG 下载器是**编程工具**，需要更底层的 USB 控制权限。

## 四、完整解决方案：配置 udev 规则

### 步骤 1：确认下载器硬件 ID
```bash
# 查看已连接的 USB 设备，找到下载器信息
lsusb
# 示例输出：Bus 001 Device 005: ID 0403:6014 Future Technology Devices International, Ltd FT232H Single HS USB-UART/FIFO IC
# 重要信息：Vendor ID = 0403, Product ID = 6014
```

### 步骤 2：创建 udev 规则文件
```bash
# 使用管理员权限创建规则文件
sudo nano /etc/udev/rules.d/99-digilent-usb.rules
```

在文件中添加以下内容（根据你的硬件 ID 调整）：
```
# Digilent/JTAG 下载器常用规则
SUBSYSTEM=="usb", ATTR{idVendor}=="0403", ATTR{idProduct}=="6014", MODE="0666"
SUBSYSTEM=="usb", ATTR{idVendor}=="0403", ATTR{idProduct}=="6010", MODE="0666"
# Xilinx 平台电缆
SUBSYSTEM=="usb", ATTR{idVendor}=="03fd", ATTR{idProduct}=="0008", MODE="0666"
SUBSYSTEM=="usb", ATTR{idVendor}=="03fd", ATTR{idProduct}=="0007", MODE="0666"
```

**规则解释**：
- `SUBSYSTEM=="usb"`：匹配 USB 设备
- `ATTR{idVendor}=="0403"`：匹配厂商 ID (FTDI)
- `ATTR{idProduct}=="6014"`：匹配产品 ID (FT232H)
- `MODE="0666"`：设置权限为所有用户可读可写

### 步骤 3：重新加载 udev 规则
```bash
# 重新加载规则，使配置生效
sudo udevadm control --reload
sudo udevadm trigger

# 重要：重新插拔下载器 USB 线
```

### 步骤 4：验证权限变更
```bash
# 查看设备权限是否已更改
ls -la /dev/bus/usb/001/005
# 期望输出：crw-rw-rw- 而不是 crw-rw-r--
```

## 五、Vivado 连接操作流程

1. **硬件准备**：
   - 开发板断电
   - 连接 JTAG 下载器到电脑
   - 开发板上电
   - 确认电源指示灯亮起
   - 确认启动模式开关拨至 JTAG 位置

2. **Vivado 连接**：
   ```
   打开 Vivado → Hardware Manager → Open Target → Open New Target
   → 连接向导中保持默认 (localhost:3121) → 手动选择检测到的下载器
   ```

3. **避免的操作**：
   - ❌ 不要使用 `sudo` 运行 Vivado（会导致文件权限问题）
   - ❌ 不要热插拔 JTAG 接口（先断电再连接）
   - ❌ 不要跳过“Open New Target”直接使用“Auto Connect”

# 1. 查看 USB 设备列表
lsusb

# 2. 查看当前用户所属组

```bash

dmesg | grep -i "ftdi\|jtag"

# 3. 运行 Vivado 硬件诊断
source /tools/Xilinx/Vivado/版本号/settings64.sh
vivado -mode batch -source 诊断脚本路径
```

## 七、常见问题排查清单

### 问题 1：规则添加后仍无效
- ✅ 确认规则文件语法正确（无多余空格或中文符号）
- ✅ 执行 `sudo udevadm control --reload && sudo udevadm trigger`
- ✅ **重新插拔 USB 设备**（必需！）
- ✅ 检查规则文件命名（数字前缀决定加载顺序）

### 问题 2：设备在设备管理器中频繁断开/重连
```bash
# 检查 USB 电源管理
cat /sys/bus/usb/devices/设备号/power/control
# 如为 auto，尝试改为 on
sudo sh -c 'echo on > /sys/bus/usb/devices/设备号/power/control'
```

### 问题 3：多用户系统下的权限问题
```bash
# 创建专用用户组（替代 0666 的更安全方案）
sudo groupadd fpga
sudo usermod -aG fpga $USER
# 修改 udev 规则：MODE="0660", GROUP="fpga"
# 注销后重新登录使组生效
```

## 八、原理总结

### 为什么需要写权限？
```
Vivado 通信流程：
1. Vivado 发送 JTAG 命令 → 需写入 USB 设备文件
2. 下载器接收并转发至 FPGA
3. FPGA 返回响应 → 下载器传回数据
4. Vivado 读取响应 → 需读取 USB 设备文件

无写权限时：第1步失败 → 后续流程中断 → Vivado 认为“无设备响应” → 报告“找不到 FPGA”
```

### 权限配置的本质
通过 udev 规则，在系统检测到特定 USB 设备时，自动将其权限设置为 `0666`（所有用户可读可写），而非默认的 `0644`（其他用户只读）。这**降低了安全限制，提升了可访问性**，是个人开发环境中的常用解决方案。

## 九、最佳实践建议

1. **连接顺序**：断电连接 → 上电 → 打开 Vivado
2. **权限原则**：优先使用 udev 规则而非 sudo
3. **规则管理**：将常用设备规则统一存放在一个文件中
4. **备份配置**：备份 `/etc/udev/rules.d/` 下的自定义规则
5. **文档记录**：记录开发板的 Vendor/Product ID，便于后续维护

## 十、扩展知识

### 1. 更安全的替代方案
对于多用户或生产环境，建议使用组权限而非 0666：
```bash
# 创建专用组
sudo groupadd jtag-users
sudo usermod -aG jtag-users $USER

# 修改 udev 规则
SUBSYSTEM=="usb", ATTR{idVendor}=="0403", ATTR{idProduct}=="6014", MODE="0660", GROUP="jtag-users"
```

### 2. 临时测试方案
如需快速测试，可使用临时权限更改（重启后失效）：
```bash
sudo chmod 666 /dev/bus/usb/001/005
```

### 3. 驱动冲突排查
如遇驱动冲突（如 ftdi_sio 占用设备）：
```bash
# 查看已加载的驱动模块
lsmod | grep ftdi

# 临时卸载冲突驱动
sudo rmmod ftdi_sio
sudo rmmod usbserial
```
