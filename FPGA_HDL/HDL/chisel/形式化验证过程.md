## 介绍
* 仿真是需要给出具体的输入信号测试案例，然后通过断言判断输出是否符合预期。
* 形式化验证是只给出断言，然后求解器会以断言为约束条件求解，断言涉及的变量包括输入输出信号和中间信号。
  * 若有断言被违反（即有解），则模块实现有误，求解器会倒推出能使断言出错的一个具体案例并输出波形
  * 若无断言被违反（即无解），则模块实现正确

## 验证过程
1. 在`ysyx-redbean/npc`下执行`mill -i Mundus.test`开始验证。然后会在npc/build.sc包含的搜索路径下的.scala文件中，找出包含断言测试的模块并编译（有extends AnyFlatSpec with ChiselScalatestTester with Formal的模块）
2. `gtkwave CacheFormalTop.bmc.vcd`查看使断言出错的波形，其生成于`ysyx-redbean/npc/out/Mundus/test/test.dest/sandbox/`

## 相关语法
下面以npc/Sub.scala为例：
1. 基本不用修改
```java
class SubFormalSpec extends AnyFlatSpec with ChiselScalatestTester with Formal {
  "Sub" should "pass formal verification" in {
    verify(new SubFormalTop(), Seq(
      BoundedCheck(1), 				// 验证多少时钟周期内违反断言，组合电路为1即可，缓存和流水线似乎20到30即可，可需要时再调
      BtormcEngineAnnotation 	// 调用模型检测器BtorMC
    ))
  }
}
```

2. 添加断言，断言信息须是英文
```java
assert(condition, cf"info + $(wire value)")
assert(condition, "info")
```

3. 条件判断
```java
when(condition) {
	assert(...)
	println(...)
}
```

4. 断言的插入位置
  * 可在被测模块(如Sub)中添加断言，此时可使用所有模块内定义的变量
  * 也可以另外实现一个测试顶层模块(如SubFormalTop)，在其内部实现被测模块，此时只能使用被测模块的IO变量，**如果用了被测模块内部变量会报错**
  * 测试顶层模块可在另一个文件中定义，以实现测试与代码的解耦，但只适用于无需使用被测模块内部变量的情况
