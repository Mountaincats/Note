## 在Makefile中执行make时变量处理的详细说明

### 1. 变量传递机制

#### a) 默认行为：变量不自动传递
```makefile
# 父Makefile
FOO = parent_value
BAR = parent_bar

all:
	@echo "Parent: FOO=$(FOO), BAR=$(BAR)"
	$(MAKE) child

child:
	@echo "Child:  FOO=$(FOO), BAR=$(BAR)"
```
运行 `make all` 输出：
```
Parent: FOO=parent_value, BAR=parent_bar
make child
Child:  FOO=parent_value, BAR=parent_bar
```
**注意**：变量是通过**环境变量**传递的，不是通过make参数。

#### b) 显式传递变量
```makefile
# 方式1：通过命令行参数
submake:
	$(MAKE) FOO=custom_value

# 方式2：通过环境变量（export）
export BAR
submake2:
	$(MAKE)

# 方式3：通过命令行和export结合
export EXPORT_VAR = exported_value
```

### 2. 你的Makefile中的情况
```makefile
@if make -s -f $@ ARCH=$(ARCH) $(MAKECMDGOALS); then
```

**这里只有`ARCH`被显式传递**，其他变量如`NAME`、`SRCS`是：
1. 在生成的`Makefile.foo`中定义
2. 不通过环境传递

### 3. `-f` 指定的巨大区别

#### 情况A：不指定 `-f`（使用默认Makefile）
```makefile
# 子make会：
# 1. 使用当前目录的Makefile
# 2. 继承父make的所有环境变量
# 3. 可能导致递归循环或错误目标
submake:
	$(MAKE)  # 使用Makefile
```

#### 情况B：指定 `-f Makefile.foo`
```makefile
# 子make会：
# 1. 使用指定的Makefile.foo
# 2. 不读取默认的Makefile
# 3. 有独立的变量作用域
submake:
	$(MAKE) -f Makefile.foo
```

### 4. 实际例子演示

#### 父Makefile（主文件）：
```makefile
# 主Makefile
SHARED = main_value
TARGET = main_target

test1:  # 不指定-f
	@echo "=== 测试1：不指定-f ==="
	@echo "父级: SHARED=$(SHARED)"
	$(MAKE) nested

test2:  # 指定-f
	@echo "=== 测试2：指定-f ==="
	@echo "父级: SHARED=$(SHARED)"
	$(MAKE) -f nested.mk nested

nested:
	@echo "错误：不该执行主Makefile的nested"
```

#### 嵌套Makefile（nested.mk）：
```makefile
# nested.mk
SHARED = nested_value
OTHER  = nested_other

nested:
	@echo "嵌套: SHARED=$(SHARED)"
	@echo "嵌套: OTHER=$(OTHER)"
```

运行结果：
```bash
$ make test1
=== 测试1：不指定-f ===
父级: SHARED=main_value
make nested
错误：不该执行主Makefile的nested  # 执行了主Makefile的nested目标

$ make test2  
=== 测试2：指定-f ===
父级: SHARED=main_value
make -f nested.mk nested
嵌套: SHARED=nested_value
嵌套: OTHER=nested_other
```

### 5. 在你的测试框架中的具体影响

```makefile
# 生成的内容
NAME = foo
SRCS = tests/foo.c
include $(AM_HOME)/Makefile
```

**关键区别：**

| 场景 | 使用的Makefile | 定义的变量 | 包含的规则 |
|------|---------------|-----------|-----------|
| 不指定`-f` | 当前目录的Makefile | 主Makefile的所有变量 | 主Makefile的所有规则 |
| 指定`-f Makefile.foo` | Makefile.foo | 只有`NAME`和`SRCS` | 通过`include`引入的通用规则 |

**正确做法（你的代码）：**
```makefile
make -s -f $@ ARCH=$(ARCH) $(MAKECMDGOALS)
```
- 使用生成的`Makefile.foo`
- 传递`ARCH`变量
- 传递原始目标（如`image`、`run`）

**错误做法（如果不指定`-f`）：**
```makefile
make -s ARCH=$(ARCH) $(MAKECMDGOALS)  # 错误！
```
- 会使用主Makefile
- 主Makefile的`$(ALL)`等变量会干扰
- 可能导致无限递归或错误目标

### 6. 环境变量 vs 命令行参数

```makefile
# 方式1：命令行参数（推荐）
make FOO=value  # 子make中FOO=value

# 方式2：环境变量
export FOO=value
make            # 子make继承FOO=value

# 方式3：混合
make FOO=cmd_value  # 命令行参数优先于环境变量
```

在你的代码中，`ARCH=$(ARCH)`是**命令行参数**，优先级最高。

### 7. 变量优先级总结

从高到低：
1. **命令行覆盖**：`make VAR=value`
2. **Makefile内部赋值**：`VAR = value`
3. **环境变量**：`export VAR=value`
4. **默认值**：Makefile预设的默认值

### 8. 为什么这样设计？

你的测试框架采用这种设计是因为：

1. **隔离性**：每个测试完全独立
2. **可重复性**：相同的命令产生相同结果
3. **灵活性**：可以单独运行任何测试
4. **简洁性**：主Makefile不包含测试具体逻辑

```bash
# 可以直接运行单个测试
make foo ARCH=x86_64
# 等价于
make -f Makefile.foo image ARCH=x86_64
```
