### 问题原因分析
你遇到的现象核心原因是 **终端的「行缓冲/字符回显」机制** 与 NEMU 模拟器的键盘事件捕获逻辑冲突，具体可以拆解为两点：

#### 1. 终端的默认行为（焦点在终端时）
当终端作为前台进程运行 NEMU 时，终端会默认开启 **字符回显（Echo）** 和 **行缓冲（Line Buffering）**：
- 你按下的按键首先被 **终端驱动** 捕获，而非 NEMU 模拟器；
- 终端会将按键字符直接输出（回显）到屏幕，同时缓存字符直到按下回车才传递给程序；
- NEMU 模拟的键盘设备（i8042 芯片）根本收不到这些按键事件，因此 `AM_INPUT_KEYBRD` 读取不到有效按键码。

#### 2. 焦点在其他界面时的特殊情况
当你将焦点切换到 NEMU 弹出的图形窗口（如 SDL 窗口）时：
- 这个窗口是 NEMU 直接创建的图形界面，绕过了终端的字符处理机制；
- 按键事件会被 SDL 等图形库直接捕获，并转发给 NEMU 的键盘设备模拟模块（`nemu/src/device/keyboard.c`）；
- 最终 `KBD_ADDR` 寄存器能正确接收到按键码，`AM_INPUT_KEYBRD` 就能正常识别。

### 解决方案（两种方式）
#### 方式 1：临时关闭终端的回显和缓冲（推荐）
在运行 NEMU 前，先执行以下命令修改终端属性，关闭回显和行缓冲：
```bash
# 保存当前终端设置，然后关闭回显和行缓冲
stty -echo -icanon
```
运行完 NEMU 后，恢复终端默认设置：
```bash
# 恢复终端默认属性
stty sane
```

#### 方式 2：修改 NEMU 启动脚本（永久生效）
如果不想每次手动执行命令，可以修改 `am-kernel/Makefile` 中运行 NEMU 的脚本，自动处理终端属性：
```makefile
# 找到 run 目标的执行命令，修改为：
run: build
	@stty -echo -icanon; \
	./build/$(ARCH)/nemu $(ARGS); \
	stty sane
```
这样每次执行 `make run` 时，会自动先关闭回显/缓冲，运行完 NEMU 后恢复。

#### 方式 3：强制 NEMU 使用图形窗口捕获按键
如果你的 NEMU 支持 SDL 图形界面，运行时添加参数强制弹出图形窗口，所有按键都会由图形窗口捕获：
```bash
make run ARCH=riscv64-nemu TEST=readkey ARGS="--show-screen"
```
（参数名可能因 NEMU 版本不同略有差异，如 `--sdl` 或 `--gui`）

### 原理补充：终端模式说明
| 终端模式       | 行为描述                                  | 对 NEMU 的影响                     |
|----------------|-------------------------------------------|------------------------------------|
| 标准模式（默认） | 行缓冲 + 字符回显，按回车才传递输入       | 按键被终端捕获，NEMU 无法识别     |
| 原始模式（-icanon） | 无缓冲 + 无回显，按键立即传递给程序       | 按键直接发给 NEMU，能正常识别     |

`stty -echo -icanon` 的作用就是将终端切换为「原始模式」，让按键事件绕过终端的字符处理，直接传递给 NEMU 进程。

### 验证方法
1. 执行 `stty -echo -icanon`；
2. 运行 `make run ARCH=riscv64-nemu TEST=readkey`；
3. 焦点保持在终端，按下任意按键（如方向键、空格）；
4. 预期结果：终端不再回显字符，而是输出按键的名称、码值和状态（pressed/released）。

### 总结
1. **核心原因**：终端默认的回显/行缓冲机制捕获了按键，导致 NEMU 模拟的键盘设备收不到事件；
2. **关键解决**：通过 `stty -echo -icanon` 关闭终端的缓冲和回显，让按键直接传递给 NEMU；
3. **替代方案**：强制 NEMU 使用图形窗口运行，按键由图形库捕获而非终端。
