## diff使用

### 一、 文件标识行 (File Headers)
输出以两行开始，指明了被比较的两个文件：
```
--- before modification
+++ after modification
```
*   **`---`** 行：表示第一个文件（通常是原始文件或旧版本）。这里的 `before modification` 是文件名。在真实 `diff` 输出中，通常还会跟随修改时间戳，如 `--- file.txt 2023-10-10 10:00:00.000000000 +0800`。
*   **`+++`** 行：表示第二个文件（通常是修改后的文件或新版本）。这里的 `after modification` 是文件名。

### 二、 块标题行 (Chunk Header / Hunk Header)
这是差异描述的核心，定义了差异发生的位置和范围。格式固定为：
```
@@ -l,s +l,s @@
```
*   **`@@`** 和 **`@@`**： 块标题的起止标记。
*   **`-l,s`**： 描述第一个文件（`---` 文件）的变更位置。
    *   `l` 是起始行号。
    *   `s` 是块的长度（行数）。它表示从上下文行和删除行中看到的行数。
*   **`+l,s`**： 描述第二个文件（`+++` 文件）的变更位置。
    *   `l` 是起始行号。
    *   `s` 是块的长度（行数）。它表示从上下文行和新增行中看到的行数。

**在你的示例中：**
`@@ -17,3 +17,3 @@`
*   **`-17,3`**： 在第一个文件（`before modification`）中，变更发生在**第17行开始**，总共涉及**3行**（包括上下文和要删除的行）。
*   **`+17,3`**： 在第二个文件（`after modification`）中，变更发生在**第17行开始**，总共涉及**3行**（包括上下文和要新增的行）。
*   这表示两个文件在第17行附近的某个位置存在差异，但差异点前后的上下文是相同的。

### 三、 差异内容行 (Change Lines)
在块标题之后，是具体的行内容，每行以特定符号开头：

*   **` ` (一个空格)**： **上下文行**。两个文件中都存在的、未改变的行。它帮助定位差异发生的位置。
*   **`-` (减号)**： **删除行**。只出现在第一个文件（`---` 文件）中，在第二个文件（`+++` 文件）中被移除了。
*   **`+` (加号)**： **新增行**。只出现在第二个文件（`+++` 文件）中，是第一个文件所没有的。

**在你的示例中：**
```
" Vim5 and later versions support syntax highlighting. Uncommenting the next
" line enables syntax highlighting by default.
-"syntax on
+syntax on
```
1.  前两行以**空格**开头，是**上下文行**。说明在修改前后，这两行内容都是相同的。
2.  第三行以**`-`**开头，是**删除行**。在原始文件（`before modification`）中，第19行是 `"syntax on`（被注释掉了）。
3.  第四行以**`+`**开头，是**新增行**。在修改后的文件（`after modification`）中，对应位置变成了 `syntax on`（取消了注释）。

**总结这个变更的含义：** 用户取消了 `syntax on` 这一行的注释，使其生效。

### 四、其他常见的 `diff` 输出格式：

1.  **正常格式 (Normal Format)**： `diff file1 file2` 的默认输出。
    *   格式如：`17c17` （表示第17行有更改）、`13d12` （表示第一个文件的第13行在第二个文件中被删除）、`5a6` （表示在第二个文件的第6行后新增了内容）。
    *   可读性较差，主要用于脚本处理。

2.  **上下文格式 (Context Format)**： `diff -c`。比统一格式更早出现，包含更多上下文（默认前后各3行），格式略有不同（用 `***` 和 `---` 标识文件，用 `!`、`-`、`+` 标识变更）。

**常用选项：**
*   `-u` 或 `--unified`： 生成上述的**统一格式**，最常用。
*   `-c`： 生成上下文格式。
*   `-y` 或 `--side-by-side`： 并排格式，更直观。
*   `-r`： 递归比较目录。
*   `-i`： 忽略大小写。
*   `-w`： 忽略所有空白字符的变化。
   
### 五、核心算法原理
1. 问题转化
将两个文件视为两个文本行序列：A = [a1, a2, ..., am]和 B = [b1, b2, ..., bn]
目标：找到将 A 转换为 B 所需的最小编辑操作（增、删、改）
这等价于寻找 A 和 B 的最长公共子序列（LCS）
2. 关键概念：最长公共子序列（LCS）
LCS 是两个序列中都出现且保持相对顺序的最长子序列
非连续但保持顺序：例如 "ABCBDAB"和 "BDCABA"的 LCS 是 "BCBA"
一旦找到 LCS，差异就显而易见了：
不在 LCS 中的 A 行 → 被删除
不在 LCS 中的 B 行 → 被插入
均不在 LCS 中的 AB 行 → 被编辑(或可看为先删除后插入)

## 补充：MD5
MD5是一种广泛使用的密码哈希函数，它可以将任意长度的数据（如一个文件）转换成一个固定长度（128位，通常表示为32个十六进制字符）的“数字指纹”，也称为哈希值或摘要。

它之所以能用来判断两个文件是否完全相同，主要基于其核心设计的三个特性：

1.  **确定性**：同一个输入文件，无论何时何地计算，其MD5值必定完全相同。
2.  **雪崩效应**：输入数据哪怕只有一位比特（例如文件中的一个标点符号）被修改，计算出的MD5值也会变得截然不同、毫无关联。
3.  **抗碰撞性**（在理想情况下）：理论上极难找到两个不同的文件能产生相同的MD5值。

因此，当你计算两个文件的MD5值时：
*   **如果MD5值相同**，在绝大多数实际情况下，可以近乎肯定地认为这两个文件的内容**完全一致**。
*   **如果MD5值不同**，则可以**100%确定**这两个文件的内容**不相同**。

4. 可用于**校验文件的完整性**
```bash
# use md5sum to make sure it matches the one on the site
md5sum valgrind-3.6.1.tar.bz2
可将计算出的这个 MD5 值与软件官方网站上公布的对应文件的 MD5 值进行比对
```

> **重要提示**：由于MD5算法已被证明存在严重的安全漏洞，可以被人为制造出具有相同MD5值的不同文件（即“碰撞攻击”），因此它**绝对不适用于需要防篡改的安全场景**（如数字签名、SSL证书）。但在非安全领域，如快速校验文件传输是否完整、比较备份文件等日常场景中，它仍然是一个简单有效的工具。对于更高的安全性要求，应使用更安全的算法，如SHA-256。
