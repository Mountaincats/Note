## ABI
ABI是由“目标平台”的核心组成部分共同制定和提供的。​ 更准确地说，它是一个由处理器架构（ISA）设计者、操作系统（OS）开发者和主流工具链（如编译器）社区协同定义的稳定规范

---

现在，你的程序要在一个复杂的计算机世界里运行，这个世界里有：
*   **CPU**：负责执行积木块。
*   **操作系统**：世界的管理者和服务提供者。
*   **其他程序库**：别人已经搭好的一些功能模块，你的程序想用。

**问题来了：** 你的“积木块”怎么和CPU、操作系统、其他库的“积木块”严丝合缝地拼接在一起，共同工作？

**一个具体的例子：**

你有这样一段代码：
```c
// 假设这是一个库函数
int add(int a, int b) {
    return a + b;
}

// 你的程序里调用它
int result = add(10, 20);
```

*   **没有ABI的世界：** 你的编译器可能把参数`10`和`20`放进寄存器`R1`和`R2`，然后跳转到`add`函数。但`add`函数（可能是别人用另一个编译器编译的库）却约定要从栈上取参数。结果它拿到了错误的数据，计算完全错误，程序崩溃。
*   **有ABI的世界：** 整个系统（你的编译器、库的编译器、操作系统）都遵守同一份ABI合同。合同里写明了：“在这个ISA（如x86-64的Linux系统）上，前两个整数参数用寄存器`RDI`和`RSI`传递”。于是，你的编译器把`10`放入`RDI`，`20`放入`RSI`；`add`函数也知道从`RDI`和`RSI`里取数。双方完美协作。

---

**ABI vs. API**

这是理清概念的关键：
*   **API** 是 **源代码层面** 的接口。它规定了你的源代码里可以调用什么函数名、传什么参数类型。比如`fopen(const char *, const char *)`。你换了编译器，只要API不变，源代码通常不用改。
*   **ABI** 是 **二进制层面** 的接口。它规定了编译好的二进制机器码，如何布局数据、如何调用函数。如果两个二进制模块（比如你的程序和系统库）ABI不兼容，它们就无法正确链接或运行，即使它们用同样的API编写。

---

**总结**

**ABI是二进制程序（可执行文件、库）与操作系统、硬件以及其他二进制程序之间进行“对话”和“协作”所必须遵守的一套底层约定。** 它确保了即使你的程序和系统库是由不同编译器、在不同时间编译的，只要它们遵循同一份ABI，就能像精密齿轮一样咬合在一起正确运行。

你之所以在更新系统或某个运行库后，有时会看到“不兼容”、“无法运行”的错误，往往就是因为新版本的ABI与旧程序不兼容了。这份“数字合同”被修改了。
