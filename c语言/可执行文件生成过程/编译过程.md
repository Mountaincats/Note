## 预处理（相关命令参数详见gcc文档中Options Controlling the Preprocessor）
```bash
gcc -dM -E <pro.c> 
// 生成 <.i> 文件
// -E 运行预处理
// -d<> 一个发给 GCC 预处理器的指令，用于控制其输出内容的“调试”或信息类别
// -dM 是 -d 选项的一个特定参数，处理完所有 #define指令后，输出所有已定义的宏的名称和值
// -dD 是 -d 选项的一个特定参数，输出 #define指令的同时，也会输出正常的预处理结果（例如展开后的代码）。-dM只输出宏定义列表，不输出其他预处理后的源码
// -dN 是 -d 选项的一个特定参数，类似于 -dD，但只输出宏的名称，不输出其值
```

## 词法分析
```bash
clang -fsyntax-only -Xclang -dump-tokens a.c
// -fsyntax-only 运行预处理、解析和类型检查阶段
// -Xclang 这是一个“桥接”或“传递”选项。它告诉 Clang 驱动程序，其后的参数 <arg>（在这里是 -dump-tokens）不是驱动程序本身的选项，而是要直接传递给底层的 Clang 编译器前端（Frontend）​ 或 LLVM组件的选项
// -dump-tokens 通过 -Xclang 传递给 Clang 前端的核心指令。它要求编译器在完成词法分析（Lexical Analysis）​ 后，将识别出的所有词法单元（Tokens）列表输出到标准错误（stderr）
```

## 语法分析
按照C语言的语法将识别出的token组织成树状结构, 从而梳理出源程序的层次结构, 从文件, 函数, 到语句, 表达式, 变量等. 若遇到语法错误(如漏了分号), 则报告错误. 语法分析的结果通常通过抽象语法树(Abstrace Syntax Tree, AST)的方式呈现
```bash
clang -fsyntax-only -Xclang -ast-dump a.c
// -ast-dump 输出抽象语法树（AST），用于语法和语义分析阶段
```

## 语义分析
语义分析的工作是按照C语言的语义确定AST中每个表达式的类型. 在这个过程中, 相容的类型将根据C语言标准进行类型转换(如算术类型提升). 而对于不符合语义的情况, 则报告错误. 一些符合语法但不符合语义的情况包括未定义的引用, 运算符的操作数类型不匹配(如struct mytype a; int b = a + 1;), 函数调用参数的类型和数量不匹配等

对于clang来说, 在输出AST的时候就已经给出表达式的类型了. 事实上, 大多数编译器并没有将语法分析和语义分析严格分开

语义分析的一个重要应用是**静态程序分析**, 它是指在不运行程序的情况下对源代码进行分析, 本质是分析AST中的语义信息. 这种通过静态程序分析报告代码中潜在问题的工具称为lint工具.

类似地, 也可以调用clang的lint工具来分析上述程序
```bash
clang a.c --analyze -Xanalyzer -analyzer-output=text
// --analyze 启用 Clang 的静态分析器。它不会生成可执行文件，而是让分析器模拟程序执行路径，查找潜在的逻辑错误、内存问题、安全问题等
// -Xanalyzer 表示后面的参数 -analyzer-output=text 不是传递给 Clang 前端，而是直接传递给底层的静态分析器引擎
// -analyzer-output=text 指定分析器输出报告的格式为 “文本”
```

## 中间代码生成
中间代码是一种由编译器定义的, 面向编译场景的ISA, 也称中间表示(Intermediate Representation, IR)或中间语言(Intermediate Language). 它是一种与源语言和目标机器都无关的、低层级但带类型信息的代码形式, 用于在编译过程中进行优化和代码生成

`clang`用的中间代码叫`LLVM IR`，`gcc`用的中间代码叫`GIMPLE`

可以通过如下命令来查看clang生成的中间代码:
```bash
clang -S -emit-llvm a.c
// 汇编文件 <.s>
// 文本格式的 LLVM 中间表示文件 <.ll>
// 二进制格式 LLVM 中间表示文件 <.bc>(Bitcode)
cat a.ll
// -S 运行至 LLVM 生成和优化阶段并生成目标机器的汇编代码
// -emit-llvm 指示 Clang 编译器在编译过程中生成文本格式的 LLVM 中间表示文件，而不是直接生成目标机器的汇编代码或可执行文件
```

## 编译优化
优化后的程序应满足"程序可观测行为"的一致性，一些优化技术如下：
* 常量传播 - 如果一个变量的取值是常数, 可以将该取值代入到引用处, 若代入后形成常量表达式, 可直接计算出该表达式的值
* 死代码消除 - 对于不可达(unreachable)的代码或不再使用的变量, 可将其移除
* 消除冗余操作 - 对于那些没有被读出就被覆盖的赋值操作, 可将其移除
* 代码强度削减 - 用简单的运算操作替代复杂的运算操作
* 提取公共子表达式 - 对于多次计算的子表达式, 可用中间变量保存其结果, 并在后续代码中直接引用该结果, 无需重复计算
* 循环不变代码外提 - 对于每次循环结果都一样的代码, 可以将其提到循环之前进行一次计算
* 函数内联 - 对于较小的函数, 可以在调用处将其展开, 节省函数调用的开销

>针对程序性能的优化等级: `-Ofast` > `-O3` > `-O2` > `-O1` > `-Og` > `-O0`(默认)
>针对代码大小的优化等级: `-Oz` > `-Os` > `-O1` > `-O0`(默认)

## 目标代码生成
目标代码生成的工作是将优化后的中间代码翻译成目标代码, 也即处理器相关的ISA
可以向`clang`提供编译选项`--target=xxx`来生成对应的汇编代码, 例如, 可以通过`--target=riscv64-linux-gnu`来生成riscv64的汇编代码
在生成目标代码的过程中, 编译器还会进行与目标ISA相关的优化, 相关知识参考**编译原理**

## 汇编
将汇编代码转变成指令的二进制编码. 这就是汇编(assemble)这一步骤的工作, 完成这一工作的工具称为汇编器, 它将汇编代码中的文本指令逐条翻译成相应的二进制编码, 从而得到目标文件(object file)
```bash
clang -c a.c
// 生成 <.o> 文件
// -c 运行至汇编阶段
```
利用`binutils(Binary Utilities)`工具包中的`objdump`工具来解析目标文件, 从目标文件中重新解析出汇编代码的过程称为"反汇编"(disassemble)
```bash
objdump -d a.o
riscv64-linux-gnu-objdump -d a.o
llvm-objdump -d a.o //可以自动识别目标文件对应的ISA架构
// Display assembler contents of executable sections
```

## 链接
将多个目标文件合并成最终的可执行文件, 目标文件会和已有的C程序运行时环境相关的目标文件进行合并

## 执行
现代操作系统中有一个称为"加载器"的特殊程序, 它的工作是将其他可执行文件从外存读入(加载)到内存, 并跳转到相应的程序入口处执行指令. 
加载器在加载程序的过程中会检查程序的指令序列属于哪种ISA, 如果与当前处理器的ISA不一致, 加载器将停止加载并报错.
